## 一、核心定论（可直接同步团队 & ClaudeCode）

1. **Demo 当前过渡期维持 sqlite-vec 不变**  
   现有调优、测试阶段数据规模小，sqlite-vec 完全满足检索需求，无需中途切换底层向量库，不打乱当前 Demo 闭环验收进度。

2. **P2 ZVEC 正式改造阶段统一切换为阿里 Zvec**  
   废弃此前「底层 sqlite-vec、上层 ZVEC 抽象」旧方案，重构 `ZvecEngine` 统一抽象封装层。上层对外名称仍保留「ZVEC 知识库」对外宣传口径。底层依赖阿里开源 `@zvec/zvec` 嵌入式向量库，全程不 fork 源码、不手写 C++ 内核，仅做 TS 薄封装隔离，切换成本极低。

3. **核心收益对比**  
   - 原 sqlite-vec 短板：仅 IVF 索引、无原生标量过滤、无 WAL 崩溃恢复、无 SIMD 加速、混合检索需多轮拼接，时空闸门过滤只能上层代码二次过滤，性能差  
   - 阿里 Zvec 优势：内置 HNSW/多索引、RaBitQ 量化、原生 SQL 式标量 filter，**时空区位、时间范围过滤可下沉至向量库检索层**，五级闸门 2 校验逻辑大幅简化，自带冷热分段、崩溃 WAL、跨平台原生 Node 绑定，单机支持亿级向量存储

4. **两层落地路线严格区分**  
   - 短期 Demo：sqlite-vec 过渡，仅做功能验证  
   - 正式发行版：上层 ZVEC 抽象接口不变，底层替换为阿里 zvec 包，上层业务代码几乎无改动

5. **否决早期 fork 剥离方案**：fork 会带来 C++ 编译、上游版本合并、持续维护高额成本，薄封装纯 TS 抽象层为最优低风险路线

## 二、同步给 ClaudeCode 标准回复（可直接转发）

收到这份《Zvec 源码研究与天权适配分析》完整调研结论，调研逻辑清晰，对比维度完整，落地路径合理，全部采纳，统一调整工程规划：

1. 现阶段 Demo 调优阶段继续沿用 sqlite-vec，不做底层向量库替换，保证 Demo 全链路验收不受干扰；
2. P2 阶段 ZVEC 改造工作调整底层依赖方案：对外架构名称仍统一叫 ZVEC 向量知识库，内部新增 `ZvecEngine` 抽象封装层，底层切换阿里开源 `@zvec/zvec` 嵌入式向量库，完全不 fork、不修改 zvec 底层 C++ 源码，仅通过 TS 封装隔离；
3. 核心优化落地：将五级闸门 2 的区位指纹、时间窗口时空过滤下沉至 Zvec 原生 filter 检索条件，合并语义稠密向量 + 关键词稀疏向量单次混合查询，减少上层多轮检索拼接开销，利用原生 Segment 冷热分段实现记忆冷热分层存储；
4. 接口兼容要求：`ZvecEngine` 对外 CRUD、检索抽象接口和原 sqlite-vec 封装完全对齐，上层 M4 融合引擎业务代码无需大规模重构；
5. 中长期预留拓展：待 DNA 遗传编码规范定稿、系统稳定后，再评估是否需要深度定制内核，现阶段仅薄封装使用官方 npm 包，最小化开发与维护成本。

## 三、同步更新两份核心文档修订要点

### 《WenStarOS 天权三体全域白皮书》修改点

1. ZVEC 知识库底层实现描述替换：删除「底层基于 sqlite-vec 扩展」，改为「上层 ZVEC 统一抽象调度层，底层采用阿里巴巴开源 in-process 嵌入式 Zvec 向量引擎，无需独立向量服务，Windows/Linux/嵌入式全平台原生兼容」；
2. 新增产品差异化优势：内置 HNSW 高性能索引、RaBitQ 向量压缩、时空标量检索下沉、原生冷热数据自动分段，单机支撑亿级私有知识与仿生记忆向量。

### 《WS-ARCH-20260710 工程蓝皮书》修改点

1. 存储 P2 阶段任务更新：废弃 sqlite-vec 长期方案，新增 `ZvecEngine` 抽象层开发任务，依赖 `@zvec/zvec` 官方 Node 绑定；
2. 检索流程改造：M4 并行记忆/知识库检索时，时空区位、时间约束条件直接传入 Zvec 原生 filter 参数，简化五级闸门上层过滤计算量；
3. 存储分层规范更新：依托 Zvec 原生 Segment 机制实现记忆冷热分区，替代自研冷热管理逻辑；
4. 开发红线补充：禁止 fork、修改阿里 Zvec 底层 C++ 源码，仅允许上层 TS 封装适配；
5. 过渡期说明：Demo 环境 sqlite-vec 为临时过渡方案，正式发行版统一切换 Zvec。
