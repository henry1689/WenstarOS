# 天权底座架构设计规范 V1.0

> **状态**: 最终定稿 · 天权工程不可变基础设施
> **日期**: 2026-07-12
> **适配**: 大一统架构 V1.0 + DNA V2.0 + Master-Harris V1.0
> **定位**: 天权的四层不可变基础设施——之上的业务模块可增删改，这四层不得变更

---

## 序言 · 什么是底座

天权不是一个模块、不是一个进程、不是一套代码。天权是**硅基生命体的智脑**——三体中唯一拥有记忆、推理、决策、生长能力的域。

天权的架构分为两层：

```
┌──────────────────────────────────────┐
│            业务层 (可变)              │
│  Master-Harris / 三库 / 知识库 /      │
│  人物图谱 / M6-M9 / 工作流 YAML       │
│  → 可以增删改、替换、扩展              │
├──────────────────────────────────────┤
│            底座层 (不可变)             │
│  第一层: DNA 编码总线                 │
│  第二层: 双螺旋存储                    │
│  第三层: 五级闸门                      │
│  第四层: 双核推理                      │
│  → 动了任何一层 = 硅基生命体基本能力改变  │
└──────────────────────────────────────┘
```

底座层是**编译期常量级的不可变基础设施**。业务层是**运行时可插拔的功能模块**。这个区分是整个天权工程最基础的架构原则。

## 第一章 · 第一底座：DNA 编码总线

### 1.1 定位

DNA 编码总线是天权的**唯一数据入口**。任何数据——对话、体感快照、工程报告、知识条目——在进入天权任何存储层之前，必须经过 DNA 编码。

### 1.2 核心约束

```
🔴 DNA 编码是天权所有数据的唯一入口，不可绕过
🔴 编码过程是同步的、阻塞的——DNA 生成完毕前数据不得进入任何存储层
🔴 主干生成后永久只读——新增信息通过追加功能分支实现
🔴 GlobalUID 是唯一的身份标识——任何模块不得自建 ID 生成逻辑
```

### 1.3 接口契约

所有数据进入天权时，调用方必须提供：

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `raw_input` | string | 用户/Master-Harris | 原始文本内容 |
| `entity_genes` | EntityGene[] | M1 L3 编码器 | 实体标注结果 |
| `leaf_zone` | LeafZone | M1 L2 编码器 | 语义区分类 |
| `scene_tags` | string[] | DNAEncoder | 场景标签 |
| `location_fingerprint` | bytes(16) | 瑶光→Master-Harris 注入 | 128-bit 区位指纹 |
| `spec_version` | string | Master-Harris 注入 | 域规范版本号 |

DNA 编码总线返回：

| 字段 | 类型 | 说明 |
|------|------|------|
| `global_uid` | string(23) | 全球唯一标识符 |
| `l0_code` | string(4) | L0 分类码 |
| `branch_id` | string | 分支路由 ID |
| `seq_pos` | number | 会话序号 |
| `crc32` | string | 完整性校验码 |

### 1.4 与业务层的关系

```
所有业务模块 → [写入] → 必须调用 DNA.encode() → 返回 FullDNA → 写入存储底座
所有业务模块 → [读取] → 必须通过五级闸门 (第三底座) → 不直接操作存储
```

业务模块永远不直接操作 `state_spines`、`atom_address_timeline`、`memories` 三张表。写入走 DNA 编码，读取走五级闸门。

## 第二章 · 第二底座：双螺旋存储

### 2.1 定位

双螺旋存储是天权的**唯一数据持久化层**。三张表、三种索引策略、三个独立的物理存储池。

### 2.2 三座存储底座

```
底座 A · 语义向量池
  表: state_spines
  内容: 32D float32 稠密向量, 每海胆 32 行记录
  主键: (global_uid, dimension_id)
  索引: HNSW 图索引 (网状、非线性)
  用途: 自由联想 / 情绪共振 / 语义相似度匹配
  硬约束: 禁止在此层做时序排序

底座 B · 寻址治理池
  表: atom_address_timeline + atom_repair_index
  内容: GlobalUID + 时序 + 藤蔓组 + 路由戳 + 区位指纹 + CRC
  主键: global_uid (TEXT)
  索引: B+Tree (时序) + 倒排索引 (藤蔓组/实体/时间分片)
  分片: 按自然月分片 ("2026-07"), 单分片深度 ≤ 3 层
  硬约束: 禁止在此层存储语义向量

底座 C · 原始数据层
  表: memories + conversations (沿用 M2 schema)
  内容: 对话原文 / 壳肉语义单元 / L0-L3 编码产出物
  用途: 兜底溯源
  硬约束: 不建语义索引, 不做直接语义检索
```

### 2.3 数据流向

```
写入路径:
  DNA 编码 → 完整海胆对象
    ├→ 32D 向量 → state_spines (语义链)
    ├→ 寻址元数据 → atom_address_timeline (寻址链)
    └→ 壳肉 → memories/conversations (原始层)

读取路径:
  纯语义模式 (默认):
    用户查询 → 32D 编码 → HNSW(state_spines) → 候选海胆
    atom_address_timeline 全程不参与

  双链模式 (主动回忆):
    用户查询 → 寻址链先行 → B+Tree + 倒排过滤(atom_address_timeline)
    → 候选 GlobalUID 集 → HNSW 在子集内精排(state_spines) → 结果
```

### 2.4 存储底座的生命周期管理

| 数据状态 | 语义链 | 寻址链 | 原始层 |
|---------|--------|--------|--------|
| 热数据 (Hot, ≤30天) | HNSW 全量索引 | B+Tree 在线 | 在线 |
| 温数据 (Warm, 30天-2年) | HNSW 降权 | B+Tree 在线 | 在线 |
| 冷数据 (Cold, >2年) | 从 HNSW 移除 | B+Tree 归档 | 压缩归档 |
| 隔离 (Isolated) | 标记, 暂停索引 | state_flag=I | 标记 |
| 删除 (Deleted) | 从 HNSW 移除 | state_flag=D | 标记, 不物理删除 |

## 第三章 · 第三底座：五级闸门

### 3.1 定位

五级闸门是天权的**唯一数据读取通道**。任何从天权存储层取出数据的操作，必须经过五级闸门。没有旁路、不可绕过、不可运行时关闭。

### 3.2 架构位置

```
用户查询 → 语义编码 → [G1] → [G2] → [G3] → [G4] → [G5] → 候选海胆集
                        ↑      ↑      ↑      ↑      ↑
                      语义   时空   遗忘   意图   话题
                      初筛   过滤   衰减   区分   壁垒
```

### 3.3 五级定义

**G1 · 语义初筛**
- 算法: 查询向量 × HNSW 索引 → Top-K 候选
- 阈值: cosine ≥ 0.3
- 输出: 候选海胆集 (通常 50-200 条)
- 不可关闭: 是

**G2 · 时空一致性校验**
- 算法: 候选集逐条比对 `location_fingerprint`
- 阈值: cosine 距离
  - ≤ 0.3 → PASS: 全量加载
  - 0.3-0.6 → P1: 仅保留 L2/L3 摘要, 屏蔽原始场景
  - 0.6-0.8 → P2: 全局权重降至 < 0.3
  - > 0.8 → P3: 直接剔除
- 瑶光空白期降级: 区位全 0 → 弱匹配 (打印告警, 不拦截)
- 不可关闭: 是

**G3 · 仿生遗忘**
- 算法: `decayed = originalWeight × exp(-λ × elapsedHours)`
- 参数: λ=0.01 常态 / 0.03 环境切换 / floor=0.05
- 三级衰减 (每 24h): 强情感(-0.02) / 工作(-0.05) / 中性(-0.10)
- 不可关闭: 是

**G4 · 意图区分**
- 算法: regex 检测时间/人物/地点关键词
- 主动回忆: 完整记忆加载
- 被动闲聊: 仅加载摘要
- 不可关闭: 是

**G5 · 话题壁垒**
- 算法: 话题切换时, 前话题的记忆链冻结
- 语义聚类 + 切换检测
- 不可关闭: 是

## 第四章 · 第四底座：双核推理

### 4.1 定位

天权使用两种完全不同的推理方式。**BIOS 核和 Mind 核**不是两个模块——它们是两种本质上不同的思考方式。

### 4.2 双核对比

| 维度 | BIOS 核 (实时核) | Mind 核 (推理核) |
|------|-----------------|-----------------|
| 本质 | 反射神经 | 意识 |
| 速度 | 毫秒级 | 秒级 |
| LLM 调用 | **永不** | **依赖** |
| 确定性 | 100% 确定 | 非确定 |
| 职责 | DNA 编码 / 32D 校验 / 安全阈值判定 / 五级闸门过滤 / 一致性仲裁 | 语义理解 / 上下文组装 / 回复生成 / 复杂推理 / 策略选择 |
| 运行频率 | 每次交互必执行 | 按需执行 |

### 4.3 双核之间的铁律

```
🔴 BIOS 核永不调用 LLM
🔴 Mind 核永不直接操作存储 (所有读写经 BIOS → 存储底座 / 五级闸门)
🔴 BIOS 核执行优先级高于 Mind 核
🔴 BIOS 核判定危险/违规时, Mind 核的 LLM 调用被直接阻断
```

### 4.4 双核的协作流程

```
用户输入
  │
  ▼
BIOS 核 (同步, 阻塞)
  ├── DNA 编码 (M1 L0-L3, 12 节点)
  ├── 32D 语义刺生成 (瑶灵×瑶光对偶计算)
  ├── 安全阈值判定 (危险=直接拒绝)
  ├── 五级闸门过滤 (G1-G5)
  └── 一致性仲裁 (瑶灵主观 vs 瑶光客观基线)
  │
  ├── 纯闲聊/低钙化 → 跳过 Mind 核, BIOS 直接返回简易应答
  │
  ▼
Mind 核 (异步, 按需)
  ├── 上下文组装 (M4 检索结果 + 知识库融合 + 人物图谱)
  ├── 策略选择 (M5 StrategySelector)
  ├── LLM 生成 (DeepSeek API)
  └── 后处理校准 (HumanisticCalibrator + SceneAnchor 一致性校验)
  │
  ▼
回复输出
```

## 第五章 · 底座层接口契约

### 5.1 第一底座 → 第二底座

```
DNA.encode(rawInput, baseData, branchExts) → FullDNA
  ├→ Storage.writeSemanticSpines(fullDNA.semanticVector) → state_spines
  ├→ Storage.writeAddressTimeline(fullDNA.addressStruct) → atom_address_timeline
  └→ Storage.writeRawFlesh(fullDNA.rawText, fullDNA.entityGenes) → memories
```

### 5.2 第二底座 → 第三底座

```
Retrieval.query(queryVector, filters) → 必须先经 FiveStageGate
  ├→ Gate1.semanticPreFilter(queryVector, topK) → candidates
  ├→ Gate2.spatiotemporalCheck(candidates, currentLocation) → filtered
  ├→ Gate3.forgettingDecay(filtered) → weighted
  ├→ Gate4.intentDistinction(weighted, userIntent) → selected
  └→ Gate5.topicBarrier(selected, currentTopic) → finalResults
```

### 5.3 第三底座 → 第四底座 (Mind 核)

```
BIOS 输出 (过滤后的海胆集, 钙化分, 安全判定)
  → Mind 核接收
  → CognitionAssembler 组装认知上下文
  → StrategySelector 选择生成策略
  → LLM 生成回复
  → HumanisticCalibrator 后处理
```

### 5.4 Mind 核 → 第二底座 (写回)

```
Mind 核生成回复后:
  ├→ 不直接写存储
  ├→ 通过 BIOS 核 → DNA 编码 (追加路由戳) → 更新 state_spines (钙化) + atom_address_timeline (路由戳追加) + memories (对话记录)
  └→ 所有写操作由 BIOS 核统一代理
```

## 第六章 · 底座层自身的容灾

### 6.1 语义链故障

```
语义链 HNSW 索引损坏 → 降级为纯寻址链检索 + 关键词匹配
→ 语义联想能力丧失, 但精确回溯能力保留
→ 通过 atom_repair_index 修复散落 32D 向量
```

### 6.2 寻址链故障

```
寻址链 B+Tree 索引损坏 → 自动切换为纯语义检索模式
→ 时序回溯能力丧失, 但自由联想能力保留
→ 任一分片损坏不影响其他分片 (自然月分片隔离)
```

### 6.3 单海胆损坏

```
单颗海胆的某根语义刺损坏:
  → 标记 consistency_mark = 'isolated'
  → 用同维度历史均值替代
  → 打印告警, 不阻塞检索

单颗海胆的寻址链断裂:
  → 通过 atom_repair_index 查找 spine_storage_position + flesh_storage_position
  → 重建寻址链记录
  → repair_count += 1
```

### 6.4 双链全部不可用

```
极端场景: 语义链 + 寻址链同时损坏
  → 降级为纯 LLM 无记忆应答
  → 打印 FATAL 告警
  → 不中断用户对话 (这是最基础的底线)
```

## 第七章 · 底座层开发铁律

```
🔴 四个底座是天权的不可变基础设施——改动任何一层即视为架构级变更
🔴 业务层模块 (Master-Harris/三库/知识库/M6-M9/YAML) 可增删改, 但不能绕过底座接口
🔴 业务层永远不直接操作 state_spines / atom_address_timeline / memories
🔴 写入必经过 DNA 编码 (第一底座)
🔴 读取必经过五级闸门 (第三底座)
🔴 LLM 调用仅限于 Mind 核 (第四底座 BIOS 隔离)
🔴 底座层之间通过严格的接口契约通信, 不跨层直接访问
```
