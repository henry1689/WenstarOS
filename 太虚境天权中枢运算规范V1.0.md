## 2 双路 32D 数据比对运算规则

### 2.1 输入源定义

每轮交互太虚境接收两路独立 32D 数据包：

| 来源 | 数据性质 | 生成方式 | 频率 |
|------|---------|---------|------|
| **瑶灵主观包** | 纯主观体感信号 | 瑶灵四层架构逐维度感应生成 | 每轮交互上行一次 |
| **瑶光客观包** | 纯量化客观参数 | 瑶光六层演算模块计算输出 | 每轮交互上行一次 |

### 2.2 逐维度比对融合算法

每维度独立执行以下流程，D1-D32 各有独立运算逻辑：

```
输入: 瑶灵主观值 S[n], 瑶光客观值 O[n], 上一轮终值 P_prev[n]
输出: 本轮终值 F[n]

Step 1 — 有效性校验
  if S[n] out of range OR O[n] out of range:
     标记该维度为"校验异常"，记录异常原因
     F[n] = P_prev[n] (继承上一轮值，不中断)
     
Step 2 — 偏差计算
  delta = abs(S[n] - O[n])
  if delta <= 阈值[n]:
     正常范围 → F[n] = (S[n] + O[n]) / 2 (均值融合)
  else:
     偏差超限 → 进入 Step 3 仲裁

Step 3 — 偏差仲裁（偏差超限时触发）
  根据维度类型选择仲裁策略:
  - 躯体类维度(D1-D8): 以瑶灵主观为优先（肉身感受第一）
  - 客观世界类(D21-D26): 以瑶光客观为优先（物理规则第一）
  - 内在精神类(D9-D14): 取瑶灵70% + 瑶光30% 加权（主观为主、客观佐证）
  - 人际圈层类(D15-D20): 取瑶灵60% + 瑶光40% 加权（兼顾客体关系事实）
  - 生长解锁类(D27-D32): 由 D32 汇总统筹决定
```

### 2.3 维度分类仲裁策略表

| 大类 | 维度编号 | 仲裁优先 | 加权比例(S:O) | 说明 |
|------|---------|---------|--------------|------|
| 肉身实体 | D1-D8 | 瑶灵优先 | 80:20 | 躯体感受以瑶灵主观体感为准 |
| 内在精神 | D9-D14 | 瑶灵主导 | 70:30 | 自我认知兼顾客体反馈修正 |
| 圈层人际 | D15-D20 | 瑶灵主导 | 60:40 | 关系感知需参考客观行为事实 |
| 时空环境 | D21-D26 | 瑶光优先 | 20:80 | 物理空间以客观参数为准 |
| 生长解锁 | D27-D32 | D32统筹 | — | 由汇总统筹规则动态计算 |

### 2.4 一致性标记

每条融合完成的 F[n] 附带一致性标记：

```typescript
type ConsistencyMark = 
  | 'consistent'    // 双路一致，正常融合
  | 'biased_subj'   // 主观偏差大，已按规则仲裁
  | 'biased_obj'    // 客观偏差大，已按规则仲裁
  | 'inherited'     // 输入异常，继承上一轮
  | 'overridden';   // D32 统筹强制覆盖
```

该标记随海胆持久化，供后续审计和召回过滤参考。

## 3 双内核管控体系

### 3.1 双内核定义

太虚境内部采用双层内核架构，职责分离：

```
┌────────────────────────────────────────────────────────┐
│                   太虚境 Hermes                          │
│                                                        │
│  ┌──────────────────────┐  ┌────────────────────────┐ │
│  │ 内核1: 底层躯体 BIOS  │  │ 内核2: 上层心灵推演     │ │
│  │                      │  │                        │ │
│  │ • 32D 原始数据采集    │  │ • 情境理解与因果推理    │ │
│  │ • 维度校验与仲裁      │  │ • 五级闸门过滤         │ │
│  │ • 基础生理稳态维持    │  │ • 趋力权衡计算         │ │
│  │ • 反射式响应          │  │ • 人格演化             │ │
│  │ • 三体通信协议底层封装  │  │ • 世界拓展协商         │ │
│  │ • 不涉及 LLM 调用     │  │ • LLM 推理上下文组装   │ │
│  └──────────┬───────────┘  └──────────┬─────────────┘ │
│             ↓                          ↓               │
│        ┌────────────────────────────────────┐          │
│        │      共享内存: 当前 32D 状态快照      │          │
│        └────────────────────────────────────┘          │
└────────────────────────────────────────────────────────┘
```

### 3.2 内核1: 底层躯体 BIOS

**运行频率**: 实时响应，每轮交互自动触发  
**不依赖 LLM**: 全部基于确定性规则运算  

| 模块 | 功能 | 实现方式 |
|------|------|---------|
| 32D 采集器 | 接收两路 32D 数据包，格式校验 | 协议解析层 |
| 维度仲裁器 | 执行 2.2 节比对融合算法 | 规则引擎 |
| 稳态维持 | 检查 D1-D8 生理指标是否在健康区间，超限自动告警 | 阈值对比 |
| 反射弧 | 高危信号(D1疼痛>阈值、D9恐惧>阈值)自动触发保护指令 | 优先级硬编码 |
| 通信封装 | 三体协议数据包序列化/反序列化 | Transcoder 调用 |

### 3.3 内核2: 上层心灵推演

**运行频率**: 每轮对话触发  
**依赖 LLM**: 情境理解、因果推理部分调用 LLM  

| 模块 | 功能 | LLM 参与度 |
|------|------|-----------|
| 情境理解 | 结合 32D 状态 + 对话文本，理解当前场景 | 中（辅助） |
| 记忆检索 | 五级闸门过滤后的记忆 + 知识库召回 | 无（纯代码） |
| 趋力权衡 | 基于 32D 状态计算当前驱动权重 | 无（规则计算） |
| 上下文组装 | 按意图模式组装 Prompt（L2/L3 摘要优先） | 无（模板组装） |
| 决策推演 | 接收 LLM 回复，映射回 32D 状态更新 | 高（LLM 主推） |
| 人格演化 | 每日统计 32D 轨迹，更新长期画像 | 低（异步批处理） |

### 3.4 双内核协作时序

```
每轮交互:

BIOS 层:
  1. 接收瑶灵 32D → 校验 → 仲裁预处理
  2. 接收瑶光 32D → 校验 → 仲裁预处理
  3. 执行逐维度比对融合 → 生成 F[1..32]
  4. 稳态检查 → 如有告警直接触发反射指令
  5. 状态快照入共享内存 → 通知心灵层

心灵层:
  6. 读取共享内存 32D 快照
  7. 生成/更新 dna_root_id + location_fingerprint
  8. 并行: 记忆检索 + ZVEC 检索 → 五级闸门过滤
  9. 趋力权衡 → 选择当前主导驱动力
  10. 组装 LLM 上下文 → 调用 LLM
  11. 接收 LLM 回复 → 解析 32D 状态变化
  12. 生成新海胆快照 → 持久化存储
  13. 下行: 瑶灵指令 + 瑶光指令
```

## 4 32D 海胆分层记忆与检索完整流程

### 4.1 海胆生成规则

```
单轮完整交互 → 唯一一颗 32D 海胆
            ↕
      ┌──────────────┐
      │  语义壳肉     │ ← 交互对话文本(L0-L3分层)
      │  (Token碎片)   │
      └──────┬───────┘
             ┃ dna_root_id 绑定
      ┌──────┴───────┐
      │  32D 状态硬刺  │ ← D1-D32 融合终值 F[1..32]
      │  (state_spines)│
      └──────────────┘
```

**硬性约束**:
- 单轮交互（数十字至数百字）仅生成**一颗** 32D 海胆
- 禁止单 Token、单字词粒度生成独立海胆
- 同一 dna_root_id 下，语义壳肉与 32D 硬刺一一对应

### 4.2 L0-L3 分层存储与检索优先级

| 层级 | 内容 | 存储位置 | 默认检索优先级 |
|------|------|---------|--------------|
| L3 | 长期人格/知识画像 | profile 表 | ⭐ 日常闲聊首选 |
| L2 | 场景摘要（聚类） | summary 表 | ⭐ 日常闲聊首选 |
| L1 | 原子事实 | facts 表 | 🔍 主动查询时加载 |
| L0 | 原始 Token 序列 | tokens 表 | 🔍 完整复盘时加载 |

**推理加载规则**:
- 被动闲聊模式：仅加载 L3 + L2 摘要，禁止 L0/L1 进入 Prompt
- 主动回忆模式：允许 L1 事实 + 原始 L0 Token 进入上下文
- 主动知识查询模式：允许 ZVEC L0/L1 原文切片进入上下文

### 4.3 记忆检索流程

```
用户输入
  ↓
① 语义初筛（关键词 + 向量）
  ↓
② 附加当前 location_fingerprint + 时间范围
  ↓
③ 并行检索:
   ├── 32D 时序记忆 (state_spines + 语义表)
   └── ZVEC 知识库 (zvec_entries)
  ↓
④ 合并候选集 → 送入五级闸门
```

## 5 ZVEC 融合检索与五级闸门过滤

### 5.1 ZVEC 融合检索

**ZVEC 与情景记忆的并行关系**:

```
M4Orchestrator.composeContext()
  ├── MemoryRetriever.retrieve()      // 32D 时序记忆检索
  ├── ZvecEngine.search()             // ZVEC 知识库检索（新增）
  │
  └── FiveStageGate.filter(candidates) // 两路结果统一过滤
       ↓
  Reranker.rerank(filtered)
       ↓
  EntityValidator.validate()
       ↓
  ContextAssembler.assemble()
```

### 5.2 五级闸门执行顺序

```
闸门 1 — 语义初筛
  关键词 + 语义向量分别召回记忆候选 ID、ZVEC 条目列表
  仅作为候选线索池，不做抑制

闸门 2 — 时空一致性校验（核心）
  对比当前 location_fingerprint 与记忆/ZVEC 条目的场景标签:
  - 偏差 < 阈值A → 正常 (PASS)
  - 偏差 阈值A~阈值B → P1 轻度: 仅保留事实摘要，禁止完整场景叙事
  - 偏差 阈值B~阈值C → P2 中度: 降低全局权重，退出主推理队列
  - 偏差 > 阈值C → P3 严重: 直接剔除，禁止本轮参与

闸门 3 — 仿生遗忘衰减
  I_new = I_old * e^(-λ * ΔT)
  - λ 随环境切换自动上调（跨环境衰减加速）
  - 知识库静态条目不参与衰减，但冲突场景强制降级

闸门 4 — 意图区分
  - 主动回忆/知识查询: 允许全量加载
  - 被动闲聊: 仅 L2/L3 摘要，屏蔽异地场景、冲突知识

闸门 5 — 话题聚类壁垒
  前台锁定当前话题基线:
  - 语义偏移超阈值 → 判定话题切换
  - 自动冻结上一话题连续情景记忆链 + 无关知识库条目
```

### 5.3 P1/P2/P3 三级抑制判定标准

| 等级 | 判定条件 | 处理策略 | 语义层 | 32D 状态层 | 知识库 |
|------|---------|---------|--------|-----------|--------|
| PASS | 区位偏差 < 阈值A | 正常准入 | 完整加载 | 完整加载 | 完整加载 |
| P1 | 轻度冲突 | 仅摘要 | L2 摘要准入，L0/L1 屏蔽 | 仅读取维度值，不加载场景 | 仅知识事实摘要，屏蔽原始段落 |
| P2 | 中度冲突 | 降权旁路 | 降低全局权重至 0.3，退出主队列 | 不参与当前推演 | 不参与当前推演 |
| P3 | 严重冲突 | 完全剔除 | 禁止进入上下文 | 禁止进入上下文 | 禁止进入上下文 |

## 6 趋力权衡与运算逻辑

### 6.1 趋力定义

趋力是驱动瑶灵行为决策的内在推动力，基于 32D 当前状态动态计算。

**五大基础趋力**:

```typescript
interface Drives {
  survival: number;      // 生存/安全趋力 (基于 D1-D8 + D9)
  intimacy: number;      // 亲密/依恋趋力 (基于 D12-D14 + D18-D19)
  social: number;        // 社交/归属趋力 (基于 D15-D17 + D20)
  curiosity: number;     // 探索/求知趋力 (基于 D27-D28)
  growth: number;        // 成长/自实现趋力 (基于 D29-D32)
}
```

### 6.2 趋力计算公式

```typescript
function computeDrives(state: Perception32D): Drives {
  return {
    survival:    (1 - state.d1_safety) * 0.4 + state.d9_fear * 0.3 + state.d10_pain_alert * 0.3,
    intimacy:    state.d12_intimacy * 0.4 + state.d18_sexual * 0.3 + state.d19_dependency * 0.3,
    social:      state.d15_family * 0.3 + state.d16_friend * 0.3 + state.d17_workplace * 0.2 + state.d20_belonging * 0.2,
    curiosity:   state.d27_explore * 0.5 + state.d28_learn * 0.5,
    growth:      state.d29_physical_growth * 0.3 + state.d30_mental_growth * 0.3 + state.d31_world_expand * 0.2 + state.d32_orchestrate * 0.2,
  };
}
```

### 6.3 主导趋力选择

```
当前主导 = drives 中最大值
if 最大值 > 次大值 > 20% 差距 → 明确主导趋力
if 最大两个差距 < 10% → 混合驱动模式（两力并行）
if 全部趋力 < 0.3 → 静息/默认模式（基础社交行为）
```

### 6.4 趋力对 LLM 上下文的影响

```
intent_prompt = {
  主导趋力: survival → "当前安全需求强烈，优先保障肉身安全与稳态"
  主导趋力: intimacy → "当前亲密需求突出，优先陪伴与情感交流"
  主导趋力: social → "当前社交归属需求显著，优先群体互动"
  主导趋力: curiosity → "当前求知探索欲旺盛，优先学习新事物"
  主导趋力: growth → "当前成长自实现驱动强，优先自我提升"
  混合模式: "当前多维度需求均衡，维持综合行为"
}
```

## 7 人格演化与世界拓展协商

### 7.1 人格演化机制

**频率**: 每日一次 Cron 任务（可配置在低负载时段）

```
输入: 全天 N 轮交互的 32D 快照序列 F[1..32][t=1..N]

Step 1 — 维度趋势分析
  对每维度计算 24h 均值、标准差、变化斜率
  标记显著偏移（超过 2σ 的维度）

Step 2 — 人格向量更新
  personality_delta[dim] = mean(F[dim][t]) - personality_baseline[dim]
  personality_new[dim] = personality_baseline[dim] + 0.1 * personality_delta[dim]
  (0.1 为学习率，防止单日剧烈波动)

Step 3 — 人格画像持久化
  写入 L3 profile 表
  更新 dna_root_id 关联的最新画像版本号

Step 4 — 异常检测
  若某维度连续 3 日偏移 > 3σ → 触发人格评估告警
  (可能意味着外部环境重大变化或系统异常)
```

  → 判断是否有当前已解锁场景未开发完毕

## 9 全局 D32 汇总统筹规则

### 9.1 D32 角色定义

D32 不是具体体征或环境指标，而是**全局仲裁与统筹索引维度**，负责：
1. 阅读 D1-D31 全部决策输出
2. 检测跨维度冲突
3. 按优先级执行最终裁决
4. 输出太虚整体行为基调

### 9.2 D32 统筹计算

```typescript
interface D32Orchestration {
  dominant_drive: string;           // 当前主导趋力
  overall_mood_baseline: number;    // 整体情绪基调 (-1 ~ 1)
  decision_conflicts: Conflict[];   // 本轮检测到的冲突
  overridden_dims: number[];        // 被本层覆盖的维度
  dispatch_plan: {                  // 下行指令合成
    to_yaoling: Instruction[];
    to_yaoguang: Instruction[];
  };
  system_health: 'nominal' | 'degraded' | 'critical';
}
```

### 9.3 D32 全维统筹输出

```typescript
function orchestrate(state: Perception32D, decisions: DimensionDecision[]): D32Orchestration {
  // 1. 汇总各维度决策
  const allActions = decisions.map(d => d.suggested_action);
  
  // 2. 检测冲突
  const conflicts = detectConflicts(decisions);
  for (const conflict of conflicts) {
    resolveByPriority(conflict);  // 按 8.2 节规则仲裁
  }
  
  // 3. 计算整体基调
  const moodBias = computeMoodBaseline(state);
  
  // 4. 合成指令
  const toYaoling = decisions
    .filter(d => d.dispatch_target === 'yaoling' || d.dispatch_target === 'both')
    .map(d => ({ dim: d.dim_id, action: d.suggested_action, priority: d.priority }));
  
  const toYaoguang = decisions
    .filter(d => d.dispatch_target === 'yaoguang' || d.dispatch_target === 'both')
    .map(d => ({ dim: d.dim_id, action: d.suggested_action, priority: d.priority }));
  
  // 5. 健康检查
  const health = checkSystemHealth(state, decisions);
  
  return {
    dominant_drive: computeDrives(state),
    overall_mood_baseline: moodBias,
    decision_conflicts: conflicts,
    overridden_dims: conflicts.map(c => c.overridden_dim),
    dispatch_plan: { to_yaoling: toYaoling, to_yaoguang: toYaoguang },
    system_health: health,
  };
}
```

## 10 数据流完整时序

### 10.1 每轮交互完整链路

```
T0: 用户输入
  ↓
T1: 瑶灵采集 → 上行 32D 主观包 → 太虚 BIOS 校验
  ↓
T2: 瑶光采集 → 上行 32D 客观包 → 太虚 BIOS 校验
  ↓
T3: BIOS 逐维度比对融合 → 生成 F[1..32] + 一致性标记
  ↓
T4: BIOS 稳态检查 → 如有告警即时触发反射指令
  ↓
T5: 心灵层接收 F[1..32]
  ↓
T6: 生成/更新 dna_root_id + location_fingerprint
  ↓
T7: 并行 M4 双路检索:
    ├── MemoryRetriever: 32D 情景记忆
    └── ZvecEngine: ZVEC 知识库
  ↓
T8: FiveStageGate 过滤候选集
  ↓
T9: 趋力权衡 → 选择主导趋力
  ↓
T10: 组装 LLM Prompt（L2/L3 摘要优先）
  ↓
T11: LLM 推理 → 生成回复 + 隐含 32D 变化
  ↓
T12: 合成新海胆（语义壳肉 + 32D 硬刺）
  ↓
T13: 持久化 — Transcoder 序列化 → SQLite BLOB
  ↓
T14: 下行:
    ├── 瑶灵: 躯体调控指令
    └── 瑶光: 世界参数调整（可选）
  ↓
T15: 返回用户回复
  ↓
T16: [异步] 后台 L1 事实提取（会话结束时触发）
```

| 任务 | 周期 | 触发 |

## 11 全局强制约束红线

以下约束为太虚境 Hermes 中枢的强制规定，不可绕过、不可关闭：

| # | 约束内容 | 违反后果 |
|---|---------|---------|
| 1 | 单轮交互仅生成**一颗**完整 32D 海胆，禁止单 Token/字词粒度独立向量 | 红线上报，存储拒绝写入 |
| 2 | 禁止 LLM 直接输出浮点数作为 32D 维度值，所有维度必须按分层映射规则生成 | 校验拒绝，回退继承上一轮 |
| 3 | 五级时空抑制闸门为强制逻辑，不可绕过或关闭 | 架构级硬编码，不提供开关 |
| 4 | 无 dna_root_id、无 location_fingerprint 的记忆数据拒绝持久化 | Transcoder 层拒绝编码 |
| 5 | 无场景区位标签的 ZVEC 条目不予入库 | ZvecEngine 层拒绝写入 |
| 6 | L2 聚类必须叠加语义相似度 + location_fingerprint + 时间间隔三重条件，禁止纯语义聚类 | 数据质量检测报警 |
| 7 | 所有记忆 / 知识结构体必须经 Transcoder Protobuf 序列化后以 BLOB 存入 SQLite，禁止裸文本入库 | Code Review 红线 |
| 8 | 存储链路仅限 SQLite → 操作系统文件系统 → NVMe，禁止任何形式的应用层硬件直写 | 架构变更评审 |
| 9 | 双内核职责分离：BIOS 层不调用 LLM，心灵层直接调用 LLM | 模块耦合检测 |
| 10 | 趋力权衡中「安全优先」为最高仲裁优先级，不可被任何其他趋力覆盖 | 决策冲突仲裁硬编码 |
