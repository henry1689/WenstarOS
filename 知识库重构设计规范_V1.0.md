## 序言 · 知识库和记忆库是两套系统

记忆库（三库体系：砂金→金库→黑钻）存储的是**生命体的自传记忆**——"我经历了什么"。知识库存储的是**生命体拥有的知识**——"我知道什么"。

两者的核心区别：

| 维度 | 记忆库 | 知识库 |
|------|--------|--------|
| 内容 | 对话体验、情感经历 | 文档、规范、事实、技能 |
| 来源 | 用户交互产生 | 用户上传 / 系统生成 |
| 时间属性 | 有明确的发生时间 (海胆 GlobalUID) | 知识本身无时间性 (但绑定来源记忆) |
| 衰减 | 钙化衰减、遗忘 | 不衰减 (仅冷热归档) |
| L0-L3 | 记忆蒸馏 (chat.ts pipeline) | 文档分层 (Wiki 化管线) |
| 检索方式 | HNSW 语义 + 寻址链时序 | Zvec 混合检索 (语义+关键词+场景过滤) |
| 产品入口 | 无感陪伴推送 (主) | Vault 可视化面板 (辅) |

## 第一章 · 当前状态与目标状态

### 1.1 当前实现 (wenstar-cc)

| 组件 | 技术 | 限制 |
|------|------|------|
| 嵌入引擎 | TF-IDF 256D (本地词级) | 无语义理解, 仅关键词匹配 |
| 向量存储 | 纯内存 Float64Array + 线性扫描 | < 10K 文档, 不持久化 |
| 混合检索 | RAGPipeline: 向量+关键词+情感 | 三路独立查询, 非一次调用 |
| 文件解析 | 8 类格式 + OCR | 音频/视频仅元数据 |
| 知识分层 | Taixu Library: raw → wiki → schema | 独立进程 :3737 |
| 场景绑定 | `scene_tags` 字段 + Jaccard 相似度 | 无向量化场景过滤 |

### 1.2 目标状态 (P2 阶段)

| 组件 | 技术 | 收益 |
|------|------|------|
| 嵌入引擎 | Zvec HNSW + RaBitQ 量化 | 语义检索, 32×内存压缩 |
| 向量存储 | Zvec 持久化 Segment (WAL + 磁盘) | 亿级文档, 崩溃恢复 |
| 混合检索 | Zvec 单次调用: 语义+稀疏+标量+FTS | 一次查询, 四路融合 |
| 文件解析 | 复用现有 8 类解析器 | 不变 |
| 知识分层 | 统一到太虚境主进程 (废弃 :3737 独立进程) | 简化部署 |
| 场景绑定 | `location_fingerprint` + Zvec `filter` 下推 | G2 闸门直接过滤 |

## 第二章 · Zvec 集成方案

### 2.1 为什么选 Zvec

Zvec (阿里通义 Lab 开源) 是**进程内嵌入式向量数据库**——"向量数据库界的 SQLite"。它与天权的需求完全对齐：

| 天权需求 | Zvec 对应 |
|---------|---------|
| 32D float32 向量 | `VectorSchema(dim=32, fp32)` |
| 区位指纹过滤 (G2) | `filter="location == ? AND timestamp BETWEEN ? AND ?"` |
| 语义 + 关键词 + 场景三合一 | 稠密向量 + 稀疏向量 + 标量过滤 + FTS = 一次调用 |
| 冷热数据分层 | Segment 冷热管理 + 内存索引 |
| 崩溃恢复 | Zvec 原生 WAL |
| 内存受限 | RaBitQ 量化: 32×内存压缩 |

### 2.2 薄封装策略

```
不 fork、不改 C++、不维护上游

天权 TS 侧:
  ZvecEngine 抽象层 (TypeScript)
    → @zvec/zvec N-API 原生绑定
      → libzvec C++ 内核 (阿里维护)

接口不变原则:
  ZvecEngine.search() 的签名无论底层是 Zvec 还是 sqlite-vec 都不变
  P2 切换时只替换实现, 调用方零修改
```

### 2.3 Collection 设计

```
Collection 1: knowledge_semantic
  ├── 向量: 32D float32 (由 Zvec 从文档内容提取)
  ├── 标量: scene_tags, classification, source_type, interaction_type
  ├── 稀疏: 中文分词关键词
  └── 用途: 语义检索 + 场景过滤 + 分类筛选

Collection 2: knowledge_fulltext
  ├── FTS: 标题 + 正文全文索引
  └── 用途: 精确关键词搜索 + BM25 排序

Collection 3: knowledge_embeddings (暂不启用)
  ├── 高维嵌入: bge-small-zh 512D
  └── 用途: 更高精度的语义重排序 (P2 阶段可选)

检索流程:
  1. Zvec 混合查询: 32D 向量 + 场景过滤 + 关键词 = 候选集
  2. RRF (Reciprocal Rank Fusion) 融合排序
  3. 返回 Top-K
```

## 第三章 · 三源融合引擎 (FusionEngine)

### 3.1 定位

三源融合引擎是天权的**上下文组装中枢**。在一次用户交互中, Mind 核需要三个来源的信息来生成回复：

```
用户查询
  │
  ▼
┌─────────────────────────────────────────┐
│          三源融合引擎                      │
│                                           │
│  知识库内容 + 记忆片段 + 人物图谱上下文       │
│       │           │           │             │
│       ▼           ▼           ▼             │
│   加权排序 → 去重 → 截断 → 注入 LLM prompt    │
└─────────────────────────────────────────┘
```

### 3.2 可信度层级

融合引擎按可信度对信息来源排序——高可信度的内容优先保留, 低可信度的在截断时先被移除。

| 层级 | 来源 | 可信度分 | 说明 |
|------|------|---------|------|
| 10 | 黑钻珍藏记忆 | 10 | 用户标记为重要 / 钙化极高 |
| 7 | 用户曾提到的记忆 | 7 | 金库中的活跃记忆 |
| 6 | 性格画像 | 6 | MasterProfile / FamilyGraph 档案 |
| 5 | 核心解答 | 5 | 知识库中精准匹配的问题-解答对 |
| 5 | 关联知识 | 5 | 知识库中场景/情感匹配的内容 |
| 5 | 知识库补充 | 5 | 泛化匹配的知识条目 |
| 4 | 玉瑶想起 | 4 | 模糊关联的记忆 |
| 3 | 时间检索 | 3 | 按时间范围匹配的对话 |
| 3 | 内心独白 | 3 | M6 自我模型输出 |
| 3 | 用户状态 | 3 | M3 感知 + 钙化分 |
| 2 | 线索参考 | 2 | M5 线索助手提供 |

### 3.3 感知驱动的融合模式

```
亲密模式 (intimacy > 0.4):
  ├── 知识库过滤: 仅保留 人物档案 / 家庭规则 / 外貌 / 性格 / 职业
  ├── 记忆增强: 珍藏记忆+2, 用户曾提到+2
  ├── 注入: 最近记忆 (top-2) + 家庭上下文
  └── 决策日志: "亲密度↑，过滤知识，增强记忆"

低落模式 (pleasure < -0.2):
  ├── 知识库过滤: 仅保留 安慰/陪伴/温暖/支持 相关内容
  ├── 记忆增强: +1
  ├── 注入: 记忆 + 家庭上下文
  └── 决策日志: "低落↑，保留温柔内容"

事实模式 (factual > 0.5, intimacy < 0.3, pleasure > -0.1):
  ├── 知识直通: 不做过滤
  ├── 不注入: 记忆和家庭上下文
  └── 决策日志: "事实查询↑，仅知识库"

中性模式 (默认):
  ├── 全部通过
  └── 不额外过滤
```

### 3.4 截断策略

```
总上限: 6000 字符
截断优先级: 从最低可信度开始移除, 同层级按内容长度降序 (短内容优先保留)

去重规则:
  同一人物实体在两条不同片段中出现 → 保留可信度更高的那条
  同一知识条目被多次匹配 → 保留第一次, 标记重复
```

## 第四章 · 知识库 L0-L3 分层

### 4.1 与记忆 L0-L3 的差异

知识库和记忆库各自有 L0-L3, 但语义不同：

| 层级 | 记忆 L0-L3 | 知识库 L0-L3 |
|------|----------|------------|
| L0 | 对话原文 (token 级) | 原始文档全文 |
| L1 | 原子事实提取 (异步) | Wiki 化正文 (结构化) |
| L2 | 场景摘要 (批次) | 语义摘要 (100-200 字) |
| L3 | 人格/知识画像 (每日) | 关键词 + 实体 + 关系索引 |

### 4.2 文件摄入管线

```
文件上传
  │
  ▼
[格式解析] txt/md/pdf/docx/xlsx/csv → 纯文本
  图片 → Tesseract OCR
  音频/视频 → 仅提取元数据 (文件名/大小/时长)
  │
  ▼
[分块] FileChunker: paragraph/heading/fixed/line 四种策略
  chunkSize=500, overlap=50, minChunkLen=20
  │
  ▼
[DNA 编码] M1 L0-L3 管道: locus_path → scene_tags → entity_genes
  │
  ▼
[入库] ZvecEntry
  l0_raw = 原始文本
  l1_facts = 异步 Wiki 化 (去 Markdown 语法 + 保留关键结构)
  l2_summary = 批次 100-200 字摘要
  l3_profile = 关键词 + 实体 + 分类标签
  │
  ▼
Zvec Collection (向量索引) + SQLite knowledge_base (结构化字段)
```

### 4.3 分类体系

```
5 大分类:
  architecture  → 白皮书 / 蓝皮书 / 架构 / 设计 / 模块
  spec          → 规范 / 标准 / 编码 / 契约 / 接口
  plan          → 计划书 / 里程碑 / 甘特图 / 开发
  audit         → 审计 / 稽核 / 检查 / 合规
  record        → 记录 / 日志 / 报告 / checkpoint

分类规则:
  标题关键词匹配 + 内容特征检测
  未分类标记 classification_pending=true
  3 天后仍未分类 → 系统自动提示用户
```

## 第五章 · 场景绑定与 G2 闸门联动

### 5.1 场景标签体系

每个知识条目入库时自动绑定场景标签：

```
来源 1: DNA L0 locus_path → 硬编码映射 (21 条)
  例: user.work.project → ["工作", "开发"]

来源 2: 文档内容关键词检测
  例: 文档中出现 "PostgreSQL" → 追加 "数据库"

来源 3: 瑶光客观环境参数 (瑶光建成后)
  location_fingerprint 作为精确区位绑定
```

### 5.2 G2 闸门过滤

```
知识库检索时, G2 闸门通过 Zvec 的标量 filter 下推:

  # 语义搜索 + G2 时空过滤 (一次 Zvec 调用)
  results = zvec.search(
    vector=queryVector,
    filter="location_fingerprint = ?",
    filter_params=[currentLocation],
    topK=20
  )

无需先检索再过滤 → 过滤在 Zvec 内部完成, 候选直接从亿级缩到百级
```

## 第六章 · 双模式产品设计

### 6.1 主线：无感陪伴知识推送（默认）

```
触发机制:
  ├── 话题切换 + 瑶灵 32D 感知 + 区位信号
  ├── 知识库匹配当前场景/心境的知识碎片
  └── 在聊天中自然带出, 不显式提示 "根据知识库..."

零操作负担:
  ├── 无需手动打开面板
  ├── 无需手动分类打标
  └── 无需主动检索文件
```

### 6.2 支线：Obsidian 风格 Vault 窗口（手动唤起）

```
7 个必须实现的组件:
  FileTree     → 树形文件浏览器
  TabEditor    → 多 Tab Markdown 编辑器
  WikiLink     → [[双链]] 解析 + 双向关联
  GraphView    → 知识图谱可视化
  Dataview     → 标签/时序 SQL 筛选
  Canvas       → 拖拽画布 + 思维导图
  Preview      → 多模态预览 (表格/代码/图片/音视频)

约束:
  ├── 不做第三方插件生态 (仅内置原生功能)
  ├── 复杂批量操作统一交 警幻仙姑 后台执行
  └── 与主线共享同一套 Zvec + L0-L3 数据, 双向同步
```

### 6.3 双向同步

```
前端 Vault 手动编辑 → onSave → 实时写入 Zvec 元数据 → 触发向量增量更新
后台自动生成摘要/双链 → onComplete → 同步渲染至前端图谱/标签面板
关闭窗口 → onClose → 自动保存所有改动
底层强制: 禁止存在两套独立文件/向量存储
```

## 第七章 · 警幻仙姑批处理 API

### 7.1 API 清单

| # | API | 功能 | 输入 | 输出 |
|---|-----|------|------|------|
| 1 | `batchGenerateSummary` | 批量 L2 摘要 | `fileIds[]` | `SummaryResult[]` |
| 2 | `batchAutoLink` | 批量 [[双链]] 补充 | `fileIds[]` | `LinkResult[]` |
| 3 | `batchTagScene` | 批量区位标签绑定 | `fileIds[]` | `TagResult[]` |
| 4 | `canvasAutoBuild` | Canvas 画布自动生成 | `rootDir` | `CanvasNodes` |
| 5 | `batchCodeComment` | 批量代码注释 | `fileIds[]` | `CommentResult[]` |
| 6 | `tableConvert` | 表格结构化转换 | `fileIds[]` | `ConvertResult[]` |
| 7 | `vaultMigrate` | 存量文档一键迁移 | `sourceDir, targetDir` | `MigrateResult` |
| 8 | `vaultArchive` | 定时冷热归档 | `before: timestamp` | `ArchiveResult` |

### 7.2 调用约束

```
🔴 警幻仙姑仅做后台批处理 + 体检工具
🔴 不替代前端 Vault 交互窗口的手动编辑能力
🔴 所有批处理操作绑定 dna_root_id 用于审计追溯
🔴 批量操作超时: 60s (100 个文档)
🔴 失败不阻塞主流程, 结果写入 aqc_records
```

## 第八章 · 迁移路径

```
阶段 1 (当前):
  TF-IDF 256D + 内存 Float64Array + 线性扫描
  Taixu Library 独立进程 :3737
  FusionEngine 三源融合 (基于纯文本匹配 + Jaccard)

  ZvecEngine 抽象层 → @zvec/zvec

  存量知识条目批量重编码为 Protobuf BLOB

阶段 P5 (远期):
  自研原生嵌入引擎 (移除 Zvec 依赖, 100% 自控)
  多模态知识检索 (图片/音频)
```
